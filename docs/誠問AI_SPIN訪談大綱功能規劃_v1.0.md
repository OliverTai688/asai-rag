## 一、研究發現摘要

### SPIN 的核心精神（常被誤解的地方）

SPIN Selling 基於 Huthwaite International 對 **35,000 場銷售對話** 的 12 年研究。核心發現是：

> **讓客戶自己說出需求，比業務員說服更有力。提問是工具，不是腳本。**

SPIN **不是線性清單**，而是一個對話心智模型。優秀業務員根據客戶回應靈活切換問題類型，而非按順序照本宣科。

### 台灣保險業的三個特殊情境

1. **文化屏障**：台灣客戶對「談錢」和「談死亡」有迴避傾向，需要迂迴但精準的問法
2. **情感決策**：台灣保險是 B2C + 情感導向，和 SPIN 原版 B2B 大型銷售設計有落差
3. **六大風險框架**：台灣業務員普遍使用「醫療 → 重疾 → 失能 → 壽險 → 退休 → 長照」的挖掘順序

### 標準保險訪談大綱的必備結構

台灣業務員實際使用的訪談大綱包含以下區塊：

| 區塊 | 內容 | 對應 SPIN 階段 |
| :--- | :--- | :--- |
| 基本資料 | 年齡、職業、家庭結構 | S（情況） |
| 財務輪廓 | 收入、支出、負債、儲蓄空間 | S（情況） |
| 既有保障 | 現有保單種類與保額 | S（情況） |
| 風險意識 | 最擔心哪個風險、身邊的事件刺激 | P（問題） |
| 連鎖影響 | 若風險發生，對家庭的具體衝擊 | I（暗示） |
| 保障期望 | 客戶自己說出希望的解決方向 | N（需求） |
| 情感動機 | 為誰、為何、什麼事 | N（需求） |
| 訪談筆記 | 情緒反應、關鍵轉折點、原話記錄 | 全程 |
| 下一步行動 | 提案方向、下次會面重點 | 結尾 |

## 二、現有功能的問題診斷

### 問題 1：Enter 直接送出，打斷思考流

**現象**：使用者在輸入框按 Enter 就立即送出訊息。  
**影響**：業務員思考問題時容易誤觸，且無法「先打草稿再確認送出」。  
**方向**：改為 `Shift+Enter` 送出，或加入「確認送出」機制。  
> 備註：目前程式碼雖然寫了 `Enter` 送出、`Shift+Enter` 換行，但 placeholder 的說明是「Press Enter to Send」，和實際操作期待有落差，建議改為按鈕或更明確的互動提示。

### 問題 2：只有對話，沒有結構化輸出

**現象**：對話結束後，只有 COMPLETE 摘要卡片，沒有完整的「訪談大綱」文件。  
**影響**：業務員無法帶著一份有結構的問題清單去拜訪客戶。  
**方向**：SPIN 對話完成後，自動生成一份可下載、可分享的「訪談大綱」。

### 問題 3：AI 只給問題，不說明「為什麼問」

**現象**：AI 提供問題建議，但業務員不知道背後邏輯。  
**影響**：業務員在實際訪談中無法靈活調整，只能念稿。  
**方向**：每個建議問題附上「為什麼問這題」的說明。

### 問題 4：訪談流程是線性的，現實是動態的

**現象**：對話按 S→P→I→N 固定推進，客戶回應無法影響流程。  
**影響**：當客戶主動說出重要資訊，AI 無法即時抓住並調整方向。  
**方向**：AI 根據客戶最新回應，即時推薦最適合的下一個問題類型。

### 問題 5：沒有客戶輪廓面板

**現象**：對話過程中收集到的客戶資訊（家庭結構、收入、保單）分散在訊息裡。  
**影響**：業務員無法一眼看清目前已掌握的情況。  
**方向**：側邊欄即時更新客戶輪廓摘要。

## 三、功能規劃

### 功能 A：輸入體驗優化

**問題**：Enter 直接送出造成誤觸，且思考中的草稿無法修改。

**方案**：

```
改動 1：輸入框改為「Enter 換行，點按鈕或 Cmd+Enter 送出」
改動 2：placeholder 說明改為「輸入你想告訴 AI 的情況... Cmd+Enter 送出」
改動 3：輸入框下方提示文字從「Press Enter to Send」改為「↵ 換行 ｜ ⌘↵ 送出」
```

**修改位置**：`src/app/(dashboard)/spin/[sessionId]/page.tsx` 輸入框的 `onKeyDown` 邏輯與 placeholder。

---

### 功能 B：動態問題推薦列（Real-time Question Suggestions）

**目的**：根據當前 SPIN 階段與客戶最新回應，即時顯示 2-3 個建議問題，每個問題附上「為什麼問這題」。

**UI 位置**：訊息列表下方、輸入框上方，以可收合的卡片形式呈現。

**資料流**：

```
使用者送出訊息
    ↓
AI 回應（現有流程）
    ↓
串流結束後，另呼叫 /api/mock/ai/spin-suggestions
    ↓
回傳 2-3 個建議問題（含問題類型、問題文字、推薦理由）
    ↓
顯示在輸入框上方的「建議問題列」
    ↓
使用者點擊其中一個 → 填入輸入框（可再編輯後送出）
```

**建議問題資料結構**：

```typescript
interface SpinSuggestion {
  type: "S" | "P" | "I" | "N";   // 問題類型
  question: string;               // 建議問題文字
  rationale: string;              // 為什麼推薦這題（1句話）
}
```

**UI 設計**：

```
┌─────────────────────────────────────────────────────┐
│  💡 建議下一個問題                          [收合 ∧] │
├─────────────────────────────────────────────────────┤
│  [I] 如果這份收入突然中斷，家裡的房貸能撐多久？      │
│      → 客戶有房貸且單薪，量化失能衝擊最直接          │
│                                           [使用這題] │
├─────────────────────────────────────────────────────┤
│  [P] 您目前的醫療險，住院自費的部分有理賠嗎？        │
│      → 客戶只提到意外險，醫療缺口需要確認            │
│                                           [使用這題] │
└─────────────────────────────────────────────────────┘
```

### 功能 C：客戶輪廓即時面板（Client Profile Panel）

**目的**：對話過程中，右側（或可收合的側邊區域）即時顯示已掌握的客戶情況，讓業務員一眼知道哪些資訊還缺。

**資料來源**：AI 在每次回應後，除了正常對話內容，也輸出結構化的「已知情況更新」（用新的 `[[PROFILE:...]]` 標記）。

**面板結構**：

```
客戶輪廓快照
─────────────────────────
家庭結構    已掌握 ✓
職業收入    已掌握 ✓
既有保單    待了解 ○
財務負債    部分 ◑
健康狀況    待了解 ○
─────────────────────────
風險意識
  醫療       ★★★ (反應強烈)
  失能       ★★  (有提到)
  壽險       ★   (未觸及)
  退休       ○   (未觸及)
```

**MVP 簡化版**（不需要 AI 解析，僅顯示各階段 outputs）：

直接把 `session.outputs` 的四個階段各有幾筆資料，用小圖示在對話頁面的右上角顯示，讓業務員知道目前的「收穫量」。

### 功能 D：SPIN 訪談大綱生成（核心功能）

**目的**：SPIN 對話完成後，自動生成一份完整的「訪談大綱」，業務員可攜帶此大綱去正式拜訪客戶。

**觸發時機**：點擊「結束對話」→ 進入 COMPLETE → 摘要卡片上方顯示「生成訪談大綱」按鈕。

**訪談大綱的輸出內容**：

```markdown
# 訪談大綱 — 林建華 / 2026-05-15

## 客戶現況摘要
- 身份：中小企業主，年收入約 500 萬
- 家庭：（待補充）
- 既有保障：（待確認）

## 本次訪談確認的重點問題

### S 情況性問題（先掌握這些）
□ 目前家裡有幾個人？孩子幾歲了？
□ 公司每月固定支出大概是多少比例？
□ 目前有幫自己或員工規劃過什麼保障嗎？

### P 問題性問題（引發思考）
□ 如果突然生病住院三個月，公司業務誰來接？財務影響多大？
□ 您的公司收入，有多少比例依賴您個人的能力與健康？

### I 暗示性問題（放大影響，重點準備）
□ 如果您有半年無法工作，公司的員工薪水、廠商貨款還能照常嗎？
□ 這個缺口如果沒有準備，最快多久會影響到您的家庭生活？

### N 需求性問題（確認動機）
□ 如果有一個方案能確保您就算無法工作，公司和家庭都不受影響，這對您來說值得認真規劃嗎？

## 預期客戶最在意的風險（依反應強度排序）
1. 企業主失能/收入中斷風險（最高）
2. 醫療費用自費缺口（中）
3. 退休規劃（待探索）

## 本次拜訪目標
- 確認家庭結構與財務輪廓
- 引發失能風險的緊迫感（I 問題重點準備）
- 取得下次提案的授權

## 建議攜帶資料
- 失能保障試算表
- 企業主保障規劃案例
```

## 四、六大風險挖掘優先順序

基於台灣保險業實務，訪談中挖掘客戶風險的建議順序：

```
1. 醫療/住院   → 發生機率最高，最容易切入，抗拒最低
2. 重大疾病    → 財務衝擊最大，容易引發危機感
3. 失能/收入中斷 → 最被忽略，但後果最嚴重（保險規劃最大缺口）
4. 身故/壽險   → 家庭責任感驅動，需要情感連結
5. 退休/老年   → 中年以上客戶的主要長期需求
6. 長期照護    → 跨代財務連動，打開機會通常需要父母議題
```

這個順序應該反映在 AI 提問建議的邏輯中，而不是固定按照 S→P→I→N 線性走。

## 五、AI Prompt 改進方向

### 現有 Prompt 問題

目前 Mock API 的每個階段只給固定的一個回應，沒有根據客戶的實際回答調整。實際接入真實 AI 後，Prompt 設計需要：

1. **納入客戶輪廓**：把 CRM 中的客戶資料（家庭結構、職業、現有保單）全部注入 system prompt
2. **保留對話歷史**：讓 AI 看到完整對話，知道哪些資訊已掌握、哪些尚未觸及
3. **指定問題類型**：告訴 AI 目前應該推薦哪類問題（S/P/I/N），不要亂跳
4. **本土化指示**：加入台灣文化語境的提示，避免直接翻譯英文問法

### 建議 System Prompt 結構

```
你是一位台灣保險業的 SPIN 銷售教練，正在協助業務員準備客戶訪談。

## 客戶資料
{clientInfo}

## 目前 SPIN 階段
{currentPhase} — {phaseDescription}

## 已掌握的情況（S 階段）
{situationOutputs}

## 發現的問題（P 階段）
{problemOutputs}

## 訪談原則
- 台灣客戶對直接談死亡和金錢有迴避傾向，請用迂迴但精準的方式切入
- 優先從「身邊發生的事」切入，比直接問自己更自然
- 問題要具體，讓客戶能回答，不要太抽象
- 每個建議問題請說明一句話的推薦理由

## 任務
根據業務員最新輸入的情況，推薦 2-3 個最適合下一步的提問，並說明為何這樣問。
```

## 八、訪談大綱的完整問題範例庫

以下為各 SPIN 階段的台灣保險情境問題，可作為 Mock API 和真實 AI Prompt 的素材：

### S — 情況性問題（掌握現況）

```
家庭結構：
□ 「您目前家裡有幾個人？孩子幾歲了？」
□ 「家裡有在奉養父母嗎？他們現在健康狀況怎樣？」

職業收入：
□ 「您現在從事什麼工作？大概做多久了？」
□ 「收入大概是固定薪資還是浮動的？」

既有保障：
□ 「您目前有幫自己或家人規劃過保險嗎？大概是什麼時候買的？」
□ 「公司有提供什麼團保嗎？」

財務狀況：
□ 「您目前有房貸或車貸嗎？大概還要繳多久？」
□ 「每個月大概能存多少錢下來？」
```

### P — 問題性問題（引發思考）

```
醫療缺口：
□ 「您之前買的醫療險，如果住院需要自費的部分，理賠金夠嗎？」
□ 「您身邊有沒有朋友或家人，最近有遇到什麼比較大的事情？像是突然生病？」

收入中斷：
□ 「如果您突然生病，沒辦法工作三個月，家裡的財務上會有什麼影響？」
□ 「您的工作如果突然停下來，誰可以幫忙撐起家裡？」

身故保障：
□ 「您有沒有想過，如果您現在發生什麼事，您先生（太太）要怎麼辦？」
□ 「孩子的教育費用，如果您不在了，有沒有什麼準備？」
```

### I — 暗示性問題（放大影響）

```
醫療衝擊：
□ 「如果住院三個月，光是自費醫療可能就要 20-30 萬，這對您的存款影響有多大？」
□ 「這段期間不能工作，同時還要付醫療費，您估計能撐多久？」

失能後果：
□ 「假設您有半年沒辦法上班，家裡的房貸和孩子的學費，每個月還要繼續付，您覺得家裡會怎樣？」
□ 「如果這個情況發生，您最擔心的是什麼部分？」

壽險缺口：
□ 「如果您現在不在了，您先生（太太）一個人要養孩子、繳房貸、同時工作，您覺得這樣的生活是什麼樣子？」
□ 「這個缺口如果不處理，對您的家人最大的影響是什麼？」
```

### N — 需求性問題（確認動機）

□ 「如果有一個方案，可以確保您就算無法工作，每個月還是有固定收入支應家庭，這對您來說有多重要？」
