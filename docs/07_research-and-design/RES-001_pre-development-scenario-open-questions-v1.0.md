## A. 訪談 Agent 流程情境

### A1 獨立模式資訊太少時的產出下限
- 為何要先談：獨立模式（無客戶資料）若業務員講得很模糊，AI 產出的輪廓表/問題清單品質會很低。
- 預設建議：設「最低必要資訊」門檻（如至少有家庭結構＋收入概況＋現有保障三選二），未達門檻時 AI 持續追問；仍不足則產出標註「初步、待補」並列出缺哪些。
- 2026-06-19 決策：採預設。

### A2 訪談中斷與跨日續談
- 為何要先談：兩份訪綱都偏長，業務員很可能談一半離開。
- 預設建議：`InterviewSession` 支援草稿續談（resume），保留已答段落與素材。
- 2026-06-19 決策：採預設。

### A3 訪談 Agent →（一鍵）劇場 Agent 的資料映射（D12）
- 為何要先談：D12 允許「從既有資料一鍵建場」，但「客戶輪廓表 / 家庭成員」要怎麼自動變成「角色卡」需定規則。
- 預設建議：焦點客戶 + `FamilyMember` 自動生成角色卡（≤4），AI 依素材補 Big Five 與情境簽名，業務員可增刪/微調。
- 2026-06-19 決策：採預設。

### A4 「對話準備卡」與既有 `VisitPlan`（訪前規劃 domain）的關係 ★
- 為何要先談：訪談 Agent 的「對話準備卡 + SPIN 問題清單 + 資料清單」與既有 `VisitPlan` 的 objectives/spinQuestions/objections/materials **高度重疊**，不釐清會做出兩套打架的東西。
- 預設建議：訪談 Agent 產出**寫入/升級既有 `VisitPlan`**（成為其生成來源），不另立重複資料物。
- 2026-06-18 決策：核可。訪談保留素材與確認卡；`VisitPlan` 保留真正拿去拜訪的準備包。

## B. 劇場 Agent 情境

### B5 是否明示「演練目標」
- 為何要先談：練「開場破冰」「異議處理」「促成下一步」的場景設定與回饋重點不同。
- 預設建議：開場讓業務員選一個演練目標，五視角回饋對齊該目標。
- 2026-06-19 決策：採預設。

### B6 角色選擇與進出場
- 為何要先談：4 人上限怎麼挑、焦點客戶是否必在場、NPC 能否中途離場（忙碌型「我先走了」）。
- 預設建議：焦點客戶必在場；業務員從 AI 建議名單選 ≤4；允許 NPC 情境性離場/再加入（牽動 §5 知情範圍）。
- 2026-06-19 決策：採預設。

### B7 私聊戰術與外洩控制 ★
- 為何要先談：真實銷售常「先私下說服配偶，再回群聊」。研究預設私聊不外洩，但要不要給業務員「主動公開/轉述」的動作？
- 預設建議：私聊預設不外洩；提供業務員觸發的「轉述/公開」動作，被轉述內容才寫入其他 NPC 的可見範圍。
- 2026-06-19 決策：採預設。

### B8 結束與結局機制（E18）
- 為何要先談：E18 設無硬上限、業務員結束。但 NPC 要不要能促成「成交/破局」的自然結局？
- 預設建議：以業務員結束為主；NPC 可表達狀態（願意再談 / 要考慮 / 談不下去）但**不自動關閉場次**。
- 2026-06-19 決策：採預設。

### B9 NPC 事實一致性（防幻覺）★
- 為何要先談：NPC 可能編造不存在的保單/數字，破壞訓練價值與合規示範。
- 預設建議：NPC 只能引用「場景設定 + 該客戶資料」內的事實；未知資訊一律表達「不確定/沒印象」，不得杜撰。後續可綁 RAG 強化。
- 2026-06-18 決策：核可。若演練中缺必要資訊，新增旁白 NPC 以情境問題詢問；使用者可選擇略過或補充已知資訊。補充內容必須標記 fact/inference/unknown。

### B10 重玩與比較
- 為何要先談：演練的學習價值常來自「換個打法重來」。
- 預設建議：支援以相同設定複製開新場；跨場比較列 M3 之後。
- 2026-06-19 決策：採預設。

## C. 人格推論與回饋情境

### C11 Big Five 推論的最低資料量與信心呈現
- 為何要先談：對話太短就下人格判斷會誤導（研究指語言線索效果量中小）。
- 預設建議：對話過短只呈現「觀察到的線索」不下推論；推論一律附信心（高/中/低）與「可修改」。
- 2026-06-19 決策：採預設。

### C12 業務員修改人格推論後，NPC 是否即時改變
- 為何要先談：影響 persona 是否為單一真實來源。
- 預設建議：以業務員「最終確認的 persona」驅動劇場 NPC。
- 2026-06-19 決策：採預設。

### C13 五視角回饋的粒度與長度
- 為何要先談：整場總評 vs 標到某幾句，體驗與成本不同。
- 預設建議：以整場總評為主，關鍵時刻可引述對話片段；每視角 1 段精簡回饋。
- 2026-06-19 決策：採預設。

### C14 「守門的良心」即時提示的 UI（G23）
- 為何要先談：即時打斷會破壞沉浸感。
- 預設建議：非阻斷式（角落紅旗/事後彙整）；僅最嚴重項（代簽/代墊/保證獲利/吸金）即時彈出。
- 2026-06-19 決策：採預設。

### C15 跨場成長軌跡
- 為何要先談：是否把五視角回饋累積成業務員「進步報告」會影響資料模型與產品定位。
- 預設建議：先存單場回饋；跨場成長分析列後續，但資料模型先預留（feedback 可被聚合）。
- 2026-06-19 決策：採預設。

## D. 合規・倫理・成本情境（上線前重點）

### D16 真實客戶資料進劇場模擬的個資/倫理界線 ★★
- 為何要先談：把真實客戶當「被模擬對象」生成人格與對話，涉及個資與職業倫理。
- 預設建議：僅限該客戶的 owner 業務員使用；介面標註「模擬用途、勿外流」；`sensitivity` 為高敏感的客戶預設**不可**進劇場，需明確確認；不對外分享劇場逐字稿。
- 2026-06-18 決策：允許 owner 顧問輸入 reason 並同意風險後使用；org manager 仍只看彙總。

### D17 紅線誤判（false positive）處理
- 為何要先談：偵測會誤報，傷害信任。
- 預設建議：提示措辭用「可能涉及…」；業務員可標記「不適用/已澄清」，不累計為負面紀錄。
- 2026-06-19 決策：採預設。

### D18 成本與配額分級（接 subscription）★
- 為何要先談：劇場每回合 2–3× LLM 呼叫且無回合上限，成本可觀；專案已有 `subscription` domain。
- 預設建議：依訂閱方案設每月「演練場次/訪談次數/字數」配額，以 `AiUsageLog` 為計量基礎；超量提示升級。
- 2026-06-18 決策：依建議。共用 `monthlyAiQuota`，但 Theater 額外顯示 theater sessions / turns soft quota。

## E. 既有系統並存與工程節點

### E19 訪談 Agent 與現有 SPIN 並存/退場（I27）
- 為何要先談：兩個入口並存會讓使用者混淆。
- 預設建議：新流程上線後，SPIN 入口標「舊版」，穩定一個版本後下架；既有 `SpinSession` 不遷移、僅唯讀保留。
- 2026-06-19 決策：採預設。

### E20 開 M1 前的 working tree 收斂與 tsc 修復 ★
- 為何要先談：目前 git working tree 有大量未提交變更（`pilot`、`subscription`、`admin`、`docs`、`prisma/schema` 等），且上線 migration 會大動 schema；不先收斂會與 migration 衝突、也難回溯。
- 預設建議：開 M1 前先做一次「整理現況 → 修復 tsc error → 依 git status 評估可 push 的節點 → commit/push」，建立乾淨基線再開發。
- 2026-06-19 決策：採預設。開 M1（PLN-015 ITA-001）前先收斂基線。
